# Git - руководство 
## 1. Первый вход.  Начало работы

## Что же такое Git?
Git - это набор консольных утилит, которые отслеживают и фиксируют изминения в файлах. Для чего он нужен? Чтобы отследить изминения, произошедшие с проектом, со временем. Проще говоря, мы можем посмотреть как менялись файлы программы, на всех этапах разработки и при необходимости вернуться назади что-то отредактировать.

## Первые шаги:

_Настройка_

* Устанавливаем Git. Открываем терминал и запускаем команды 
 ```
 git config --global user.name `"Ваше имя"`
 
  git config --global user.email `"...@com"`
  ```

* Скачиваем программу Visuai Studio и продолжаем всю остальную магию в ней.

_Работа_
* Создаем папку на рабочем столе.
* В меню программы выбераем нужную папку.
* Открывем терминал.
* Создаем файл и задаем ему имя _name.md_
* Представляемся:  
```
 git config --global user.name "Ваше имя"
 
  git config --global user.email "...@com"
```
* Вводим команду `git init` - создаем репозиторий
  

* Сперва нужно сделать фиксацию. Данные команды являются как и первичными, при начале работы так и последущими - каждый раз, когда мы выполняем работу над файлом ( вносим изминения). Фиксация выполняеться такими командами:
```
git add <имя файла>
```
```
git commit -m " Коментарий, вид изминений, которые произошли на данном этапе работы"
```
_Данные команды можно писать и в более сокращенном варианте, а именно соеденив их в одну_ :

`git commit -a -m "Коментарий"`

_или_

`git commit -am "Коментарий"`

## Кроме отдельных случаев, но об этом дальше.

* Теперь проверяем статус работы на данный момент, пишем :

```
git status
```
## 2. Что дальше?

После каждой фиксации (сохранения) нашей работы нужно посмотреть журнал изминений, есть команда :
```
git log
```
 В данном журнале содержиться вся информация о каждом отдельном коммите, с указанием его хэш-кода, автора, списка изминений и даты, когда они были сделаны.

 Чтобы посмотреть (пролистать) журнал, необходимо использовать клавиши _стрелок_ на клавиатуре или клавишу _enter_.

 Для выхода из журнала нужно нажать клавишу _Q_

## 3. Отслеживание изменений сделанных в коммите

 У каждого комита есть свой уникальный идентификатор в виде строки цифр и букв ( Вспоминаем `git log`). Как мы уже заметили, идентификаторы очень длинные, но для работы нет необходимости их копировать целиком - первых несколько символов будет достаточно. 
 
  Чтобы увидеть разницу между двумя комитами, используется команда:
  ```
   git diff <индификатор коммита>.
```
Чтобы переключиться на нужный нам коммит, необходимо воспользоваться командой :
```
git checkout <индификатор нужного коммита>
```
Для возвращение в ветку мастер вводим команду :
```
git checkout master
```

## Игнорирование файлов.

1. Первым шагом нужно будет создать новый файл с названием :
```
.gitignore
```
_Важно написать название файла верно, чтобы сработала функция игнорирования_

2. В рабочем окне пишем название и формат файла который Git должен игнорировать.
3. Теперь мы должны зафикcировать изминения. 

_Ранее мы говорили о том, что есть моменты, когда краткая команда

 `git commit -a -m "...."` или
 
 `git commit -am "...."`
 
  может сработать не верно. Это именно этот случай._
 
 Выполняем команду фиксирование изминений верно. Пишем следущее команды:
 ```
 git add.gitignore
 git commit -m "...."
 ```
 4. А так же не забываем проверить сработала ли данная функция и была ли фиксация данного действия в журнале, вспоминаем команды:
 
 `git status` и `git log`.

## 5. Дополнительная информация.
Дополнительные команды и функции.

* Используем функцию автосохранения.

* Такая команда покажет краткий вид журнала изминений :
```
git log --oneline
```
* Чтобы клонировать существующий репризиторий Git из любого метста воспользуемся командой
```
git clone <индификатор>
```
* Чтобы исправить предведущий коментарий коммита есть команда:
```
git commit --amend - m "Исправленный коментарий"
```
* Чтобы очистить терминал просто вводим команду :
```
clear
```
## 6. Создание веток
1. Для начала выполнив команду 
`git branch` мы убедимся, что находимся на основной ветке master(чистовик проекта)
2. Чтобы создать новую ветку:
```
git branch <название ветки>
```
3. Теперь с помощью уже знакомой нам команды  
`git branch` проверяем наличие новой ветки.
4. Чтобы переместиться и начать работу с веткой, необходимо так же воспользоваться уже знакомой нам командой `git checkout` вот так:
``` 
git chekcout <Название ветки>
```
5. Теперь снова проверяем на какой ветке мы находимся командой

    `git branch`
 
  Увидеть это можно двумя способами. После вызова данной команды указывает на нахождения на определенной ветке может звездочка в терминале, рядом с веткой в списке всех веток.
 Либо это можно увидеть в самой программе - внизу слева на синей панеле тоже прописывается ветка на которой мы находимся.


## 7. Работа с ветками 
* Переходим на нужную ветку ( команда перехода описана в пункте 6 ). Теперь мы спокойно можем в ней работать как в черновике. 
_Но перед тем как начинать работу, обязательно через знакомую нам команду перепроверим себя, а именно убедимся, что находимся на нужной нам ветке.
* Чтобы соеденить нужную ветку с основной веткой master, переходим на ветку master за тем выполняем команду :
```
git marge <название ветки>
```
_Не забываем после каждого видоизминения файла делать фиксацию_

`git commit -am "Инф. про выполненые изминения"`

## 8. Работа к конфликтом.
Если мы решили соеденить нашу ветку (черновик) с основной веткой(чистовик), а информация в разделе разнилась, то программа нас предупредит о так называемом конфликте. Наша нынешная и предведущая работа с которой произошел конфликт будет выделена двумя разными цветовыми областями. Сверху над ними программа предложит решение данной проблемы:

* мы можем применить текущую версию
* принять входящую версию
* принять оба варианта 
* сравнить варианты  
  
 _Чтобы посмотреть на наши все ветвления, мы можем воспользоваться командой:_

  ```
  git log --graph
  ```

_Чтобы удалить одну из веток необходимо воспользоваться командой:_
```
git branch -d <название ветки> 
```
Если возникают проблемы с удалением ветки, есть видоизмененная команда, где большая буква -D означает принудительное удаление. Посмотрим:

```
git branch -D <Название ветки>
```

## _Прирывание слияния_
Если ,вдруг, вы не были готовы к конфликтам, мы можем просто отменить попытку слияния командой :
```
git merge --abort
```
Эта команда откатит состояние до того, что было до запуска слияния. *Завершиться она неудачно если* перед запуском слияния у вас в рабочем каталоге были незафиксированые изминения, во всех остальных случаях все будет хорошо.
# Работа с удаленными репозиториями в Git
## 1.1 Отличия между локальными и удаленными репозиториями 
* Локальный репозиторий - это репозиторий который мы создаем у себя на компьютере, где мы его и сохраняем и производим дальнейшую работу с ним.
* Удаленный репозиторий - это репозиторий , с которым мы можем работать на сервере , там же выполнять дальнейшее его сохранение. 

_Если нам нужно поделиться или передать кому-то свой репозиторий мы можем как раз рассмотреть работу с удаленным репозиторием._ 
Конечно, мы можем заархивировать локальный репозиторий и отправить его, но процесс работы с удаленным репозиторием избавляет нас от выполнения лишних взаимодействий. Например, если мы работаем над проектом коллективно или вместе с другим человеком, есть возможность работать и видоизменять репозиторий через сервер.
## 1.2 GitHab - крупнейший веб-сервис
 Прежде чем описывать дальнейшую работу, разберемся чем отличаеться git от GitHab:
* git - одна из систем контроля версий
* GitHab - популярный онлайн сервис. Система управления проектами и вересиями кода, а так же платформа социальных сетей , созданная для разработчиков.

 Теперь разберемся на практике. На GitHab мы можем не только создать свой репозиторий и работать с ним коллективно, но и взять чей-то репозиторий, чтобы предложить свои изминения и идеи в чужой проект. А так же просто наблюдать за работой и видоизминением других проектов. 
## Создание удаленного репозитория на GitHab
1. Для начала нам нужно зарегестрироваться на GitHab. Заходим на сайт веб-сервиса и следуем подсказкам при регестрации, если это необходимо. 
2. Создаем у себя на компьютере папку, называем ее и открываем в Visual Studio Code.
3. Заходим на GitHab (на свою страничку аккаунта). В правом верхнем углу видим иконку с плюсиком и при нажатии на нее, выбираем команду `New repository`. 
4. При создании нового репозитория даем ему имя за тем  нажимаем кнопку _Создать репозиторий_. 
5. Чтобы продолжить работу с репозиторием на своем компьютере, мы должны обратить внимание на подсказки, которые далее в работе предлагает нам GitHab: 

* Создать новый репозиторий в командной строке
echo "# Название нашего репозитория " >> README.md 
```
git init 
git add README.md 
git commit -m "first commit" 
git branch -M main 
git remote add origin < Ссылка на удаленный репозиторий >
git push 
```

* Отправить существующий репозиторий из командной строки
```
git remote add origin < Ссылка на наш удаленный репозиторий>
 git branch -M main 
git push -u origin main
```
* Импортировать код из другого репозитория
Мы можем инициализировать этот репозиторий кодом из проекта Subversion, Mercurial или TFS.
Внизу для данного предложенного варианта есть кнопочка `Код импорта`

6. Нам подходит второй предложеный вариант.
```
При начале работы с удаленным репозиторием в Visual Studio Code, после команды push появится окошко с предложением авторизоваться от GitHab. Это нужно сделать обязательно.
```
Копируем поочередно в терминал ссылоки с предложенного нам GitHab 2-го варианта.
1-ой командой мы указываем ссылку на наш удаленный репозиторий. 2-ой командой, основной веткой для работы мы делаем ветку main. 3-й командой мы говорим что нужно "протолкнуть" наш удаленный репозиторий, для дальнейшей работы с ним в Visual Studio Code.

7. Заходим снова на нашу страничку GitHab, обновляем ее и видим наш новый репозиторий. Если мы выбрали , при создании, что наш репозиторий будет публичным ( помним, что GitHab при создании репозитория дает нам подсказки и там есть информация о выборе, каким должен быть наш репозиторий), в дальнейшем мы можем делиться ссылкой на него. Чтобы увидеть ссылку на репозиторий, нужно нажать кнопку `Code`.

8. Теперь мы можем поработать, внести информацию, закомvитить все в Visual Studio Code. Теперь, чтобы наша работа появилась на GitHab, в удаленном репозитории были видны изминения, нужно дать такую команду git
```
git push
``` 

9. Чтобы внести видоизминения в репозиторий на GitHab нажимаем иконку с карандашем, вносим необходимые изминения, следуя подсказкам при сохраненении , коммитим и все готово.
## Работа с чужими проектами
Чтобы предложить свои доработки проета другому создателю, нужно выполнить такие шаги :

1. Находим интиресующий нас проект на GitHab. Справа вверху видим иконку, напоминающую вилку с подписью `Fork`( с англ. переводится так же ). Нажимаем на нее. Высвечивается окошко, где мы можем изменить названия проекта( изменится название только у нашей вилки проекта, а не у самого автора проекта) и нажимаем кнопку создания новой вилки `Create fork`. Теперь переходим на нашу страницу и видим в списке репозиториев копию данного проекта. И уже на данном этапе мы можем скопирувать ссылку, с которой в дальнейшем будем работать.

_Почему мы не можем, таким же способом, как предведущий, просто на прямую поработать со ссылкой( копируя ее)?_ Так как автором проекта являеться другой человек, соответсвенно прямого доступа к данному проекту у нас нет.

2. Создаем папку у нас в компьютере, называем ее. Открываем ее в Visual Studio Code. Теперь вводим команду:
```
git clone <ссылка нужного репозитория>
```
Видим, что слева у нас появилась папка нужного проекта. Чтобы заглянуть в корень папки, нужно ввести команду :
```
cd <название папки>
```
Теперь мы можем посмотреть какая работа выполнялась в данном проекте командой:
```
git log --oneline --graph
```
3. Создаем новую ветку, в которой непосредственно и будем вести основную работу.

4. Важным моментом данной работы является создания нового файла. А именно его название принято писать большими буквами README.md. Все потому что именно файл с таким названием говорит о том что мы дополнительно поработали с проектом и готовы предложить свои изминения автору. Вообщем и целом принято так, значит в дальнейшем работаем именно по такому принципу. 

5. Создание веток и работа с ними нам уже известна из прошлых пунктов. Следущим шагом будет переход на нашу созданную ветку. Вся работа будет проводиться именно в ней. Можно создавать , конечно, дополнительные ветки, но мержить их можно только со своей. ( А не как в обычной работе с веткой master, в данном случае основной веткой main )

## Дальнейша работа. 
1. При клонировании файла копия всех изминений происходит в скопированом репозитории на нашей страницу Githab. 
2. После окончания работы в Visual Studio Code мы переходим на свою страницу GitHab и видем выделеное красным цветом окошко и кнопочку `Compare & pull request`, которая предлагает нам сравнить и выполнить команду pull request. Жмем на нее. GitHab предложит автоматически нашу ветку в виде кнопки справа `Compare <название нашей ветки>` И обязательно нужно обратить внимание, если есть зеленое сообщение от GitHab с галочкой `Add to merge` - это говорит нам о том, что нет проблем в выполнении дальнейшего слияния.
3. Далее нам будет предложено добавить заголовок и коментарий к нашей работе.
4. После чего нажимаем кнопку `Create pull request`. Внизу будут показаны изминения (содержимое), которые мы внесли и хотим отправить.
